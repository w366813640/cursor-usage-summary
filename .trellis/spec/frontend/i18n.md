# i18n — Internationalization

## TL;DR

- Provider is mounted in `apps/playground/src/App.tsx` with `initialLocale="en"`.
- Locale is persisted in `localStorage` under `cu:locale`.
- Two locales today: `en` (default) and `zh`.
- Dictionary lives in `packages/ui/src/i18n/dictionaries.ts`.
- Use `useT()` to read a single key, or `useI18n()` to also write the locale.
- **Do not** import a translator inside a class component — extract a function child that calls `useT()`.

## When to add a key (and when not to)

### Add a key

- Anything a user sees in the **chrome** (sidebar, headers, settings drawer, welcome hero, onboarding, keyboard cheatsheet, tooltips that always show the same text).
- Generic action verbs (`common.save`, `common.cancel`, `common.delete`, `common.close`).
- Section titles in long-lived panels.

### Skip it for now

- **Data narratives** that build sentences from rows ("$1,742 across 380 requests — $4.58/req, down 12%"). They need a locale-aware sentence builder; punting until that's prioritized. Mark them with a `// i18n-skip: data narrative` comment so they're easy to find.
- Field labels that are literally a domain term Cursor uses (`Max-mode`, `Auto pool`, model names). These read the same in every language.
- Diagnostic / debug strings (`current persisted: ...`, console errors).

## Key naming

Namespaced with dots:

```
nav.<route>
chrome.<thing>
welcome.<part>
settings.<section>[.<subkey>]
tour.<stepN>.<part>
quickTips.<thing>
cheatsheet.<thing>
trust.<thing>
common.<verb>
```

Stick with lowerCamel for the trailing segment to match the JS that consumes it. **No nesting** — flat string→string is enough and keeps the dictionary diffable.

## Placeholders

Use `{name}` placeholders, **not** ICU MessageFormat (we don't ship an ICU runtime):

```ts
'tour.stepIndicator': 'step {current} / {total}',
```

```tsx
t('tour.stepIndicator', { current: step + 1, total: STEPS.length })
```

If you need to inject a JSX child (a chip, a code span), split the template around a sentinel:

```ts
'welcome.subtitle': 'Drop in a {code} exported from cursor.com/dashboard/usage. …',
```

```tsx
const tpl = t('welcome.subtitle', { code: '<<CODE>>' });
const [pre, post] = tpl.split('<<CODE>>');
return <p>{pre}<code className="font-mono">…</code>{post}</p>;
```

Don't try to `dangerouslySetInnerHTML` the translated string — it loses the design system's chip styling.

## Adding a new locale

1. Pick a key in `Locale` (`packages/ui/src/i18n/types.ts`).
2. Add a sibling dictionary to `dictionaries.ts`, copying `enDictionary` and translating the values. Keep keys identical.
3. Register the entry in `builtInDictionaries`.
4. Add a button in `SettingsDrawer` Language section. (Three+ buttons get clunky — swap to a `<select>` at that point.)

Missing keys silently fall back to English; you don't have to translate everything to ship.

## How NOT to refactor towards translatability

- Don't move strings into a component prop just to "expose" them — that just moves the literal to the parent.
- Don't add a `defaultMessage` argument to `t()`. We can't enforce it via lint, and adding the same English twice (in the call **and** the dictionary) is a maintenance trap.
- Don't translate dynamic values you concatenate with English glue (`'Loaded: ' + fileName`). Pull the whole sentence into a single key with a placeholder.

## Common gotchas

- **Class components**: hooks aren't available. Split the rendering into a function child (see `AppErrorBoundary` → `ErrorFallback` in `App.tsx`).
- **Hooks-in-loops**: when registering N shortcuts via `useShortcut`, keep the array a stable module-level constant. Inside the loop, you can call `useT()` once at the top and reuse its result.
- **`Locale` is `'en' | 'zh'`** — both 2-character. Don't BCP-47-ify the storage key (no `zh-CN` etc) until you actually fork the dictionary.
- The provider syncs `<html lang>` automatically — never write `document.documentElement.lang = …` from a component.
