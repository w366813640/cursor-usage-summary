# i18n тАФ Internationalization

## TL;DR

- Provider is mounted in `apps/playground/src/App.tsx` with `initialLocale="en"`.
- Locale is persisted in `localStorage` under `cu:locale`.
- Two locales today: `en` (default) and `zh`.
- Dictionary lives in `packages/ui/src/i18n/dictionaries.ts`.
- Use `useT()` to read a single key, or `useI18n()` to also write the locale.
- **Do not** import a translator inside a class component тАФ extract a function child that calls `useT()`.

## When to add a key (and when not to)

### Add a key

- Anything a user sees in the **chrome** (sidebar, headers, settings drawer, welcome hero, onboarding, keyboard cheatsheet, tooltips that always show the same text).
- Generic action verbs (`common.save`, `common.cancel`, `common.delete`, `common.close`).
- Section titles in long-lived panels.
- **Data narratives** that compose sentences from rows. The pattern is
  documented under [Data narratives](#data-narratives-translator-pattern)
  below тАФ every builder takes an optional `Translator` and falls back to
  the English literal when missing.

### Skip it for now

- Field labels that are literally a domain term Cursor uses (`Max-mode`, `Auto pool`, model names). These read the same in every language.
- Diagnostic / debug strings (`current persisted: ...`, console errors).
- The exported **local markdown report** (`utils/localReport.ts`). Reports
  are artifacts users paste into issues / share with teammates тАФ keeping
  them in a single canonical language makes them grep-able. Add a `t`
  argument at the call site if you change your mind.

## Data narratives (Translator pattern)

`@cu/data` is framework-free тАФ it can't import React or `@cu/ui` тАФ so
to make narratives translatable we ship a one-line contract:

```ts
// packages/data/src/i18n.ts
export type Translator = (
  key: string,
  vars?: Record<string, string | number>,
) => string;
```

Every narrative builder accepts an optional `t?: Translator` and uses
the local `translate(t, key, englishFallback, vars)` helper. When no
translator is passed (tests, CLI, markdown export), the helper returns
the English literal тАФ same string that previously lived in the
function body, no behaviour change.

Renderer side, the React caller threads `useT()` in:

```tsx
const t = useT();
const week = useMemo(
  () => composeWeekSummary(summary, rows, { t }),
  [summary, rows, t],
);
```

Keys live in `dictionaries.ts` under the `narrative.*` namespace,
sub-grouped by builder (`narrative.weekSummary.*`,
`narrative.dayAudit.*`, `narrative.insight.*`, `narrative.anomaly.*`,
`narrative.budget.*`, `narrative.efficiency.*`, `narrative.burn.*`).
Picking a fresh sub-key for a new sentence is preferred over reusing
an existing one тАФ it keeps the dictionary searchable.

### Adding a new narrative sentence

1. Pick the sentence in the builder.
2. Replace the literal with `translate(t, 'narrative.<group>.<key>', '<english>', { vars })`.
3. Add the key to both `enDictionary` and `zhDictionary`.
4. If the builder already takes `opts`, just plumb `t` through; otherwise add an `opts?: { t?: Translator }` parameter (or a positional `t?: Translator` for single-purpose helpers like `buildDayAnswer`).

### Tests

Tests don't pass a `t`, so they continue to assert against the English
literal. If you change an English string, update the test substring at
the same time. The point of the fallback is that tests stay readable тАФ
don't add `mockT()` boilerplate just to round-trip the same string.

## Key naming

Namespaced with dots:

```
nav.<route>
chrome.<thing>
welcome.<part>
settings.<section>[.<subkey>]
tour.<stepN>.<part>
quickTips.<thing>
cheatsheet.<thing>
trust.<thing>
common.<verb>
severity.<level>                            # shared closed set (high/medium/low)
overview.<card>.<thing>                     # WeekSummary, ActionFeed, Efficiency, Burns
budget.banner.<thing>                       # urgency banner chrome (not the message body)
anomalies.<section|stat|method>.<thing>     # Anomalies page chrome
narrative.<group>.<key>                     # builder output (see Translator pattern above)
```

Stick with lowerCamel for the trailing segment to match the JS that consumes it. **No nesting** тАФ flat stringтЖТstring is enough and keeps the dictionary diffable.

## Placeholders

Use `{name}` placeholders, **not** ICU MessageFormat (we don't ship an ICU runtime):

```ts
'tour.stepIndicator': 'step {current} / {total}',
```

```tsx
t('tour.stepIndicator', { current: step + 1, total: STEPS.length })
```

If you need to inject a JSX child (a chip, a code span), split the template around a sentinel:

```ts
'welcome.subtitle': 'Drop in a {code} exported from cursor.com/dashboard/usage. тАж',
```

```tsx
const tpl = t('welcome.subtitle', { code: '<<CODE>>' });
const [pre, post] = tpl.split('<<CODE>>');
return <p>{pre}<code className="font-mono">тАж</code>{post}</p>;
```

Don't try to `dangerouslySetInnerHTML` the translated string тАФ it loses the design system's chip styling.

## Adding a new locale

1. Pick a key in `Locale` (`packages/ui/src/i18n/types.ts`).
2. Add a sibling dictionary to `dictionaries.ts`, copying `enDictionary` and translating the values. Keep keys identical.
3. Register the entry in `builtInDictionaries`.
4. Add a button in `SettingsDrawer` Language section. (Three+ buttons get clunky тАФ swap to a `<select>` at that point.)

Missing keys silently fall back to English; you don't have to translate everything to ship.

## How NOT to refactor towards translatability

- Don't move strings into a component prop just to "expose" them тАФ that just moves the literal to the parent.
- Don't add a `defaultMessage` argument to `t()`. We can't enforce it via lint, and adding the same English twice (in the call **and** the dictionary) is a maintenance trap.
- Don't translate dynamic values you concatenate with English glue (`'Loaded: ' + fileName`). Pull the whole sentence into a single key with a placeholder.

## Common gotchas

- **Class components**: hooks aren't available. Split the rendering into a function child (see `AppErrorBoundary` тЖТ `ErrorFallback` in `App.tsx`).
- **Hooks-in-loops**: when registering N shortcuts via `useShortcut`, keep the array a stable module-level constant. Inside the loop, you can call `useT()` once at the top and reuse its result.
- **`Locale` is `'en' | 'zh'`** тАФ both 2-character. Don't BCP-47-ify the storage key (no `zh-CN` etc) until you actually fork the dictionary.
- The provider syncs `<html lang>` automatically тАФ never write `document.documentElement.lang = тАж` from a component.
